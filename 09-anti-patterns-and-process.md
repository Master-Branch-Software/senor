# Anti-patterns and Process

The difference between professional and junior output rarely comes from knowing more tools. It comes from consistently avoiding a short list of common mistakes and applying a consistent review process. This chapter documents both.

## Design anti-patterns

### Hero section mistakes

- Headlines that could describe any competitor ("The future of work").
- Vague verbs on CTAs ("Learn more," "Get started" without specifying what).
- Two primary CTAs competing for attention.
- Stock photography of diverse teams staring at laptops.
- Text that sits on top of a busy image without a contrast-safe overlay or tint.
- A hero that autoplay-cycles through six different promises; users cannot read any of them.

### Typography mistakes

- Body text smaller than 16 px on any screen.
- Line height below 1.4 on body copy.
- Line length exceeding 80 characters on wide viewports.
- Three or more unrelated typefaces.
- All-caps body text.
- Light-weight (100, 200) gray text on light backgrounds. Unreadable and fails contrast.
- Using `vw` alone for font-size; text stops scaling with user preference.

### Color mistakes

- Pure `#000` on pure `#FFF`. Harsh on both light and dark screens.
- Accessible contrast traded for "aesthetic." Users should not have to squint.
- Relying on color alone to convey meaning (red = error, green = success without icon or text).
- Accent color overused so it loses emphasis.
- Dark mode generated by blind inversion, resulting in washed-out or garish results.

### Spacing mistakes

- Arbitrary `16px`, `24px`, `30px`, `42px` values that are not from a scale.
- Equal spacing around related and unrelated items, destroying grouping signals.
- Too-tight line height and too-loose letter-spacing, a common AI default.
- Forgetting that mobile screens need more generous tap targets, not tighter ones.

### Motion mistakes

- Full-page parallax scrolling, especially with text on top.
- Auto-playing videos or carousels without pause controls.
- Long entrance animations on every element on the page (users wait; they don't enjoy).
- Ignoring `prefers-reduced-motion`.
- Animating `width`, `height`, or `margin` instead of `transform`.
- Scroll-jacking — hijacking the native scroll behavior with custom JavaScript, almost always regrettable.

### Layout mistakes

- Using `100vh` for full-height mobile sections. Use `100dvh`.
- Hard-coded widths that break at unusual viewports.
- Ignoring tall, narrow viewports (foldables, split screens).
- Content trapped inside a fixed-height container that overflows.
- Flexbox children with no `min-width: 0`, causing overflow at narrow widths.

## Code anti-patterns

### HTML

- `<div onClick>` instead of `<button>`.
- `<a href="#">` used as a button.
- Nesting `<button>` inside `<a>` or vice versa.
- Placing `<section>` or `<article>` without an accessible name (no heading, no `aria-label`).
- Missing `alt` attributes.
- Structured content inside divs instead of semantic elements.

### CSS

- `!important` except in utility layers where it is explicit.
- Global styles that affect elements by tag name without a scope.
- Media queries at every breakpoint when flex-wrap or `minmax` could solve it.
- Z-index values of 999, 9999, 99999 used unpredictably.
- `position: absolute` applied to elements that should flow naturally.
- Layout properties scattered across multiple files so the cascade determines the final outcome.

### JavaScript

- `==` instead of `===`.
- Top-level `console.log` shipped to production.
- Mutating props or state directly.
- `setTimeout(fn, 0)` as a substitute for thinking about when code should run.
- Unhandled promise rejections.
- Silent `catch (e) {}` blocks.
- `innerHTML` with untrusted input.
- Storing tokens or secrets in `localStorage`.

### React / framework-specific

- `useEffect` for anything that is not a true effect (use derived state, events, or memoization).
- `useEffect` with missing dependencies to "make it work."
- Heavy components on every render, uncached.
- Unstable keys on list items (`key={index}` when the list reorders).
- Passing new function references as props on every render, defeating memoization.
- Mixing client and server state in the same store.

## AI-generated code pitfalls

Known patterns that AI assistants fall into by default. Watch for them on every review:

- Producing plausible-looking imports that do not exist in the current version of the library.
- Skipping `width`/`height` on images.
- Hard-coding pixel values for everything.
- Using `px` for font sizes.
- Ignoring `prefers-reduced-motion`.
- Producing verbose comments that narrate the code.
- Stitching together two incompatible patterns (e.g. pages router Next.js code inside an App Router file).
- Adding `aria-label` defensively, duplicating existing labels, or contradicting them.
- Inventing classes for a CSS framework that do not exist in that framework.
- Reaching for Lodash, moment, or jQuery when modern JS, CSS, or DOM APIs cover it.
- Implementing modals, menus, dialogs, and comboboxes from scratch instead of using accessible primitives (shadcn/ui, Radix, React Aria).
- Using `any` in TypeScript without narrowing.
- Missing key accessibility primitives: focus rings, labels, keyboard handlers.
- Using outdated package versions or deprecated APIs.
- Losing track of responsive behavior at intermediate viewport widths.

Every AI-generated block should be read once as a human would read a junior's first draft.

## Pre-merge checklist

Run through this list before merging any substantive change.

Correctness

- The change solves the stated problem and does not silently break existing behavior.
- Types check without `any`, without `as unknown`, without `@ts-ignore`.
- Tests cover at least the new behavior.

Design

- Uses tokens from the design system rather than ad-hoc values.
- Typography follows the scale.
- Spacing is from the scale.
- Colors meet contrast thresholds in both light and dark modes.
- Layout works at 320, 480, 768, 1024, 1440, and 1920 CSS pixels, and at 200% zoom.
- Images have `width`, `height`, meaningful `alt`, and appropriate `loading`.

Accessibility

- Every interactive element has an accessible name.
- Keyboard navigation is logical and complete.
- Visible focus indicator present on every focusable element.
- Forms have associated labels, validation messages, and proper input types.
- Dynamic content changes are announced when appropriate.
- axe or Lighthouse reports no new violations.

Performance

- No regression in Core Web Vitals.
- Bundle size stays within budget (check CI report).
- Lazy-load below-the-fold images; `fetchpriority="high"` on the LCP image.
- No new synchronous `<script>` in `<head>`.
- Third-party scripts evaluated and deferred appropriately.

Copy

- Follows the brand voice documented in `02-brand-and-copywriting.md`.
- Buttons use verbs.
- Errors name the problem and suggest a fix.
- Empty and loading states exist and are useful.

Code quality

- No commented-out code, dead imports, or debug `console.log` statements.
- Files stay in the right places; imports follow the agreed alias conventions.
- New dependencies are justified.

## Process

### From brief to launch

1. Brief — one page with audience, goal, success metric, constraints, brand voice, deadline.
2. Content outline — headings, hero copy, primary CTA, required sections.
3. Wireframes or low-fi sketches — hierarchy validated before visual work.
4. Design tokens — colors, type, spacing chosen before component work.
5. Components — build from tokens; review at real-world widths.
6. Pages — compose components; connect copy, data, CTAs.
7. Accessibility and performance passes — not a gate at the end, but a continuous activity during steps 5 and 6.
8. QA — cross-browser, cross-device, keyboard-only, screen reader.
9. Launch — staged rollout with monitoring; rollback plan ready.
10. Post-launch review — compare against the brief's success metric at 30 and 90 days.

### Design review

Every significant UI change deserves a focused review covering:

- Hierarchy — is the most important element most prominent?
- Consistency — same patterns used for the same kinds of actions?
- Brand — does it feel like this brand and no other?
- Feel — does it match the adjective chosen for the project?
- Copy — is every word doing work?
- Accessibility — keyboard, screen reader, contrast, motion.
- Performance — bundle and field metrics.

Keep reviews short (15–30 minutes) and frequent (weekly). Long, infrequent reviews produce dread and mass edits; short, frequent reviews produce incremental improvements.

### Avoiding premature scale

- Build for the next 10× of users, not 100×. Premature abstraction is expensive.
- Do not spin up microservices when a monolith suffices.
- Do not add a cache until a slow query proves one is needed.
- Do not abstract behind an interface until two implementations exist.
- Do not split a component until a second place needs it.

### Learning continuously

- Audit one well-designed site per week. Pick apart what works.
- Read the changelogs of browsers, frameworks, and standards bodies quarterly.
- Track issues and opinions from authors in the field: Chris Coyier, Josh W. Comeau, Kevin Powell, Una Kravets, Emil Kowalski, Steve Schoger, Adam Wathan, Jen Simmons, Rachel Andrew, Heydon Pickering.
- Build small experiments. Shipping small, finished things teaches more than reading.

## The shortest review

If there is time for only one question on a review: does the change make the site easier to use, easier to understand, easier to maintain, and faster? If not, why is it here?
